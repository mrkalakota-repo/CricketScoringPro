⬆ Strong Foundation

What's Good

The offline-first stance is the single most important design decision in this app, and it's executed correctly. Cricket grounds in Tier 2 and Tier 3 cities in India — and community grounds in the UK and Australia — have notoriously poor connectivity. An app that scores ball-by-ball without a signal is genuinely useful in a way that cloud-first competitors simply aren't.

The three-layer web gate (loading → marketing → auth) is architecturally clean. Showing a landing page before asking users to register reduces friction for first-time visitors and creates a proper acquisition funnel. Most devs skip this and immediately throw a signup form at visitors — you've got this right.

Standout: Auto-commentary is a genuine differentiator. "Deepak Rao castles him! Bowled for a duck" appearing instantly on every device is the kind of delight that gets shared on WhatsApp. It's a viral loop built into the scoring experience.
⚠️
What Needs Work
  • The PIN-based auth creates a confusing dual-login experience. Users have: (1) a team admin PIN, (2) a personal user PIN, and (3) OTP via SMS. That's three distinct auth concepts with no obvious hierarchy. First-time users coming from WhatsApp will be confused about which PIN goes where.
  • The web experience is significantly weaker than native. Team chat is blocked. Pro and league features throw a download prompt. The marketing landing page promises "one app, every format, everywhere" — a web user who signs in and immediately hits a wall will feel deceived.
  • The 5-country restriction is the right call for Twilio cost control, but it's invisible to users. Someone from Sri Lanka, Pakistan, or the West Indies who enters their number and gets a silent validation error will assume the app is broken.
💡
Recommendations
  • Consolidate the auth mental model into a single explainer screen on first launch: "Your phone number is your identity. Your PIN is your password. OTP lets you restore on a new device."
  • Add an explicit "Download the app for full access" CTA on web with platform-specific badges. Don't just block — convert.
  • Show the supported country list before the phone entry field, not after a failed OTP attempt. A simple "Available in: 🇮🇳 🇺🇸 🇬🇧 🇦🇺 🇳🇿" line prevents confusion.

What's Good

The market positioning is correct. ESPNCricinfo, Cricbuzz, and CricHeroes all target the professional or semi-professional tier. The gully/community segment — where teams of 11 friends play on a maidan every Sunday — is genuinely underserved. The feature set reads like it was designed by someone who has actually played gully cricket: admin PIN per team, delegate access codes, proximity discovery, 12 dismissal types. The specificity is credible.

The monetisation structure (free / pro / league) is well-conceived. League admin as a paid tier makes sense — tournament organisers derive the most value and have the highest willingness to pay.

Standout: Knockout brackets + NRR calculations make the league feature viable for serious amateur tournaments. These are the features that differentiate from "just tracking scores in a spreadsheet" — and they're already built.
⚠️
What Needs Work
  • The growth plan has a sequencing problem. You have a well-built product but no acquisition strategy. The live score broadcast (50-mile radius) is a brilliant built-in loop, but it requires the scorer's device AND the spectator's device to both have the app. The cold start problem is real — spectators won't pre-install an app they've never heard of.
  • The viewer role is underutilised. A viewer who gets a share link to a live scorecard — without having to register — is a much stronger growth driver. Make the live scorecard URL-shareable and publicly accessible without login.
  • The naming transition from "Gully Cricket Scorer" to "Inningsly" is still incomplete internally. The Android package ID is com.gullycricket.scorer — the brand should be clean externally before any public launch.
  • If RevenueCat is wired but nothing is actually gated behind a paywall, you have zero revenue signal to validate your pricing hypothesis.
💡
Recommendations
  • Build a "Share live scorecard" feature with a public URL (no login required for viewing). This turns every match into an acquisition event — the scorer gets the app, spectators get a link.
  • Define concrete paywall triggers for pro/league plans immediately. Without them, RevenueCat is sunk cost and you're flying blind on monetisation viability.
  • Secure one piece of social proof before the Play Store submission — a 10-team tournament run on Inningsly with real screenshots and user quotes. This dramatically improves store listing conversion.

⬆ Production-Grade Core
✅
What's Good

The immutable MatchEngine pattern is the correct architectural choice for a scoring engine. Every ball returns a new instance — undo is O(1), testing is straightforward, and concurrency is safe. Whoever made this call understood the domain deeply. Most sports scoring apps get this wrong and end up with a tangled mutable state blob that makes undo nearly impossible.

The concurrent write protection in publishMatchState is production-grade thinking: conditional UPDATE before upsert, using match.updatedAt (engine logical timestamp) rather than Date.now(). This prevents the classic "two scorers on different phones overwrite each other" bug that would destroy trust.

Standout: The bot protection story is layered correctly. Cloudflare Turnstile handles web clients; the country-prefix allowlist in the Edge Function is the server-side enforcement layer. Together they prevent bad actors from calling the raw Supabase function URL to spam Twilio for unsupported regions — proportionate and cost-effective.
⚠️
What Needs Work
  • The in-memory auth session reset is intentional but creates friction on web, where tab-close means session expiry. The sessionExpired flag and auto-redirect to the restore form is the correct recovery path, but the UX is jarring. sessionStorage for the PIN hash on web would survive page refreshes but clear on tab-close — identical behaviour with less friction.
  • The rate limit store diverges on cold starts. The in-memory store resets; Supabase is the source of truth. A user hitting a cold-started Edge Function could bypass the in-memory limit and only be caught by the persistent check. The effective rate limit during cold starts is 3 (persistent), not 5 (in-memory). This is by design — but it should be documented as a known trade-off, not left as a silent discrepancy.
  • Schema versioning at CURRENT_SCHEMA_VERSION = 1 is good infrastructure. But with no lint guard enforcing migrateMatch() at every JSON.parse boundary, the next developer to parse match JSON will likely forget to wrap it. This bug is silent and catastrophic — old matches silently lose fields.
  • No error monitoring (Sentry, Bugsnag, or equivalent). The app is deployed to production on Amplify and approaching app store submission. Crashes and edge-function errors are currently invisible unless users report them — the highest-leverage infrastructure gap before going public.
💡
Recommendations
  • Add Sentry (or equivalent) before the Play Store submission. The free tier is sufficient; the signal is essential. Edge Function errors and client crashes need to surface proactively.
  • Create a TypeScript branded type ParsedMatch that can only be created via migrateMatch(). Any JSON.parse → Match cast becomes a type error. One-time investment — prevents an entire category of regression.
  • Document the in-memory vs. persistent rate limit divergence with a // KNOWN: comment in the Edge Function. It's not a bug, but the next maintainer will read it as one without context.

Strengths — Consensus

  • Offline-first architecture — correct for target market
  • Immutable MatchEngine — production-grade design
  • Auto-commentary as built-in viral loop
  • Layered bot protection (Turnstile + country allowlist)
  • Concurrent write protection in cloud sync
  • League + knockout bracket feature depth
  • Clear, testable domain model (150+ tests)
  • Web entry gate + marketing landing page

Concerns — Consensus

  • Auth model complexity (3 distinct PIN/OTP concepts)
  • Web experience significantly weaker than native
  • Country restriction invisible to users
  • No growth loop — spectators need the app pre-installed
  • RevenueCat wired but no active paywall triggers
  • No error monitoring before production launch
  • Schema version guard relies on convention, not types
  • Brand name inconsistency (GullyCricket → Inningsly)
